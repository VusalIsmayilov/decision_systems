## Page: Decision Systems Design

**Role:** Category Definition  
**Stage:** Category Formation  

---

**Hero**

* Eyebrow: Decision System Design  
* Headline: A decision becomes reliable when its structure is explicit  
* Body:
  A recurring business decision can be defined as a system.

  When its structure is specified,  
  it produces consistent outcomes under the same conditions.

---

**Section**

* Section name: Structure  
* Headline: A decision system is defined through eight elements  
* Body:

1. **Decision** — The specific question being resolved, clearly bounded  
2. **Trigger** — The condition that initiates the decision  
3. **Inputs** — The exact information used for evaluation  
4. **Logic** — The rules used to interpret inputs and reach a conclusion  
5. **Ownership** — The single role accountable for the decision  
6. **Workflow** — The sequence from trigger to execution  
7. **Action** — The predefined response following the decision  
8. **Feedback** — The mechanism for tracking outcomes and refining the decision  

* Supporting elements:  
  Each element is distinct. Together, they define how the decision operates.

---

**Section**

* Section name: System behavior  
* Headline: The structure determines how the decision behaves  
* Body:
  When all elements are specified and aligned:

  * the same situation leads to the same decision  
  * the same decision leads to the same action  
  * outcomes can be explained and improved  

  When any element is implicit, variation enters the system.

## Page: Decision Systems in Practice

**Role:** Proof  
**Stage:** Diagnostic Justification  

---

**Hero**

* Eyebrow: Proof  
* Headline: Customer retention intervention as a defined decision  
* Body:
  A recurring decision: when to intervene with an at-risk customer.

  The difference is not data.  
  It is how the decision is structured.

---

**Section**

* Section name: Before  
* Headline: How the decision operates without structure  
* Body:

  * No explicit definition of “at-risk customer” (usage decline vs complaints vs inactivity)  
  * Intervention triggered inconsistently — some accounts reviewed weekly, others only after escalation  
  * Different teams use different signals (usage metrics, support tickets, account manager judgment)  
  * No consistent thresholds for when intervention is required  
  * Multiple stakeholders influence the decision without clear accountability  
  * Actions vary — some customers receive proactive outreach, others receive no intervention  
  * Outcomes are not tracked against the decision  

---

**Section**

* Section name: After  
* Headline: The same decision defined as a system  
* Body:

  * Decision defined: which customers require intervention in the current weekly cycle  
  * Trigger activated when usage drops below defined thresholds or risk score exceeds a set level  
  * Standardized input set combining product usage, support activity, and account signals  
  * Explicit logic defines intervention type based on severity and customer segment  
  * Single ownership assigned to a defined role responsible for the decision  
  * Consistent workflow from trigger → evaluation → action  
  * Predefined intervention actions (outreach, escalation, retention offer)  
  * Outcomes tracked against intervention type and timing  

---

**Section**

* Section name: Result  
* Headline: What changes when the decision is structured  
* Body:

  * The same conditions trigger the same decision  
  * Teams act consistently across accounts  
  * Interventions occur earlier and under defined conditions  
  * Outcomes are measurable and comparable across teams
